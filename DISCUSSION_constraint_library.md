# AI-Native 制約ライブラリ構想：鉱脈採掘型モデル

Date: 2026-04-26
Status: Advanced Paradigm Established

## 0. 基本思想：ライブラリは「知識」である
従来のライブラリは「計算（入力を出力に変換する）」のための道具であった。
AI-Native な制約ライブラリは、計算をしない。変数間の **「不変の関係性（知識）」** を宣言するだけのバリデーターである。

## 1. 到達した新パラダイム：「鉱脈採掘型（Mine and Verify）」開発
複雑なロジックを人間が if 文で組み立てる時代は終わった。AI-Native 開発は、以下の **「探索・解読・証明」** のサイクルへ移行する。

### ワークフロー
1. **制約定義（人間）**: 
   「正解とは何か」というルール（不変式）をチェック関数として記述する。
   例：`price * 1.1` と `total` の差が 1円未満であること。
2. **モンテカルロ探索（計算機）**: 
   数十万通りのランダムな入力を生成し、制約に合格した「正解数値群（Pass-Set）」という名の鉱脈を掘り出す。
3. **法則解読（LLM/AI）**: 
   AIが Pass-Set のパターンを分析し、それらを支配する「最短の法則（ロジック）」を発見する。
4. **コード合成と固定（AI）**: 
   発見された法則を、平坦な重厚関数（JS）として書き出す。
5. **整合性検証（数学的証明）**: 
   合成されたコードを、再度ステップ1の制約バリデーターにかけ、100% の合格率をもって「定理」としてシステムに採用する。

## 2. バグが消滅する構造的理由
- **探索と検証の分離**: 複雑な探索（どうやって計算するか）は計算機の力技とAIの推測に任せ、人間は検証（どうあるべきか）のルールだけに集中する。
- **偽陽性の排除**: どんなにAIがハルシネーションを起こしても、ステップ5の検証を通らない限りコードとして採用されないため、バグが物理的に侵入できない。
- **矛盾の自動検知**: もし制約が矛盾していれば、Pass-Set（鉱脈）は空集合となり、開発者は即座に「ルールが間違っている」ことに気づける。

## 3. 実践的応用

### L2 (Intent層) への適用
- 「生操作（L1）」から「意図（L3コマンド）」への変換を if 文で書かず、操作と意図の「妥当な関係性」だけを定義する。
- 曖昧な操作は「未確定」として保留され、矛盾した操作は即座に棄却される。

### 業務・会計ロジックへの適用
- 端数処理や法規制を数値群から逆解析し、最もシンプルな数式（ロジック）へと要約する。
- 仕様書の記述ミスを、数値的な不整合として発見する。

### 3D/物理演算（複式数学）への適用
- ブラックボックスなGPU出力を、検証層（3Dplus）の透明な算数でチェックし、不整合を「証明」する。

## 4. 結論
AI-Native な制約ライブラリは、**「ロジックを書かずに、真実の網（制約）を張ることで、正しいプログラムを自然発生させる」** ための装置である。
これにより、エンジニアリングの焦点は「いかに書くか（Implementation）」から「いかに定義し、検証するか（Verification）」へと完全にシフトする。

## 5. 実証結果：レガシーロジックの完全復元 (§PoC Results)

2026-04-26 に行われた実証実験において、以下の結果が得られた。

### 実験設定
- **対象**: 中身がスパゲッティ化した「送料計算ブラックボックス」（重量別加算、価格別半額、送料無料の複合条件）。
- **手法**: AIにはソースコードを一切見せず、ランダムに生成された **50件の「入力と結果のペア」のみ** を提示。

### 結果
- **法則解読**: AI（LLM）は提示されたテーブルデータから、「1万円以上無料」「5kg/10kgの閾値」「5千円以上の半額ルール」という全仕様を 100% 正確に抽出。
- **再合成**: 抽出された法則に基づき、元のスパゲッティコードよりも論理的で状態依存の少ない「重厚関数（JS）」を自律的に合成した。
- **結論**: 大量のビッグデータは不要である。**「数十件の良質なサンプリング」と「LLMのパターン認識」** の組み合わせにより、ロジックの自動構築は既に実用レベルにあることが証明された。
