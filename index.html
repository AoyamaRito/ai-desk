条件畳み込み

if/else のネストの代わりに「全可能世界 → 制約 filter → 結果」で導出する。データがコード、コードがデータ。逆引きが第一級になる。

// 純粋データとして遷移を宣言 const TRANSITIONS = [ { from:'JAB', input:'C', when: s => s.hit && s.frame >= 4 && s.frame <= 8, next: 'SPECIAL' }, { from:'STRONG', input:'C', when: s => s.hit && s.frame >= 8 && s.frame <= 14, next: 'SPECIAL' }, // ... ]; // 全可能世界を制約で絞るだけ。if は無い。 function resolve(state, input) { return TRANSITIONS .filter(t => t.from === state.action && t.input === input && t.when(state)) .sort((a,b) => b.prio - a.prio)[0]; } // 逆引きも一発。"SPECIAL に到達する全ルート" を全列挙。 TRANSITIONS.filter(t => t.next === 'SPECIAL'); // → 3 routes

4本の柱

AI 認知の4仮説（±300行スポットライト・近接バイアス・ファイル切替コスト・推測の弱さ）から導出される設計原則。

§0.0 · 認知の非対称性

複雑性は人の問題、隠匿は AI の問題。

人間時代の「良い設計」（DRY・抽象化・カプセル化）は人間が読みやすくするためのもの。AI に対しては、情報を物理的に分散させハルシネーションを誘発する妨害行為になる。

人間がコードを読まない時代、最適化対象は AI である。

// 04

4層バニラ

情報の流れは一方向。L3 Logic は 純粋関数として実装することが必須。これが守られないと体系全体が崩れる。

L1

Physical

DOM・イベント・外部 I/O

L2

Intent

イベント → Command 変換

L3 · CORE

Logic

(state, cmd) ⇒ newState
pure function · 必須

L4

Draw

state を狙撃描画

なぜ L3 が純粋でなければならないか: 同じ入力で常に同じ出力（網羅テスト成立）。副作用ゼロ（リプレイ可能）。出力 → 入力の逆引き可能。外部状態への依存なし（隠匿問題の解消）。これらすべてが純粋性に依存している。
// 05

業界常識への正面からの否定

人間時代のベストプラクティスの多くは、AI に対しては逆効果になる。仮説からの帰結。

✗ Antipatterns

  • アテンション分散型 TDD テスト・コードのファイル分離、目的不明の増分サイクル
  • DRY · 共通ヘルパー抽出 参照先への切替コストで認知破壊
  • 抽象化 · カプセル化 人間には複雑さの隠蔽、AI には情報の隠匿
  • SQL · RDBMS 暗黙知の塊・逆引き不能・状態破壊（四重違反）
  • 「読み手が推測する」前提のコメント省略 AI は推測できない

✓ Recommended

  • E2E テスト 目的の単一性が構造的に保証される
  • 網羅型テスト群 全可能世界を1ファイルに集約
  • JSON Event Sourcing + 直列ハッシュ 履歴不滅・暗号学的整合性・任意時点再現
  • Emblem 境界 + ai-desk 意図に対するローカリティ提供
  • 重厚関数 · インライン展開 一つの関数で完結、AI が迷わない
