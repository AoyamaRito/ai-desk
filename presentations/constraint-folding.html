Constraint Folding

複数の入力変数を、複数の状態として畳み込み、状態遷移を駆動する。
ai-desk の最深部の機構。

ai-desk 方法論 — 沖井広行 · 2026-05

github.com/AoyamaRito/ai-desk

1分岐ではなく、列挙

if/else / switch を使わない。代わりに 全可能世界をデータとして列挙し、制約で filterする。

従来 (分岐)

  • 条件を 制御フロー として書く
  • 分岐ツリーで状態を辿る
  • すべてのパスを脳内で追う必要
  • 1 箇所追加すると全体の整合性確認が必要
  • 逆引き不可 (出力 → 入力は無理)

Constraint Folding

  • 条件を データ構造 として持つ
  • filter で生き残った世界が答え
  • すべてのケースが 1 箇所に並ぶ
  • 1 行追加すれば全体の整合性は構造的に保証
  • 逆引き可能 (どの入力でその状態に行けるか)

2パターン

「全可能性を data に展開」 → 「制約で filter」 → 「残った世界が結果」

// 1. 全可能世界を列挙する (data として) const WORLDS = [ { state: 'title', input: 'start', next: 'playing' }, { state: 'playing', input: 'pause', next: 'paused' }, { state: 'playing', input: 'die', next: 'gameover' }, { state: 'paused', input: 'resume', next: 'playing' }, { state: 'gameover', input: 'reset', next: 'title' }, ]; // 2. 制約で filter する function reduce(state, input) { const world = WORLDS.find(w => w.state === state && w.input === input); return world ? world.next : state; // 該当世界がなければ現状維持 }

分岐ゼロ。データを変えるだけで状態機械が更新できる。

3複数変数 → 複数状態

入力が増えるほど分岐は爆発する。畳み込みなら データ次元が増えるだけ。

// 例: 3 変数 (画面 / 入力 / 装備) で次状態を決める const WORLDS = [ { screen: 'playing', input: 'attack', weapon: 'sword', effect: 'slash', next: 'attacking' }, { screen: 'playing', input: 'attack', weapon: 'bow', effect: 'shoot', next: 'shooting' }, { screen: 'playing', input: 'attack', weapon: 'magic', effect: 'cast', next: 'casting' }, { screen: 'paused', input: 'attack', weapon: '*', effect: null, next: 'paused' }, // ... ]; function resolve(s, i, w) { return WORLDS.find(x => x.screen === s && x.input === i && (x.weapon === w || x.weapon === '*') ); }

3 変数 × N 値の組合せが テーブル 1 つに収まる。新しい武器を足す = 行を 1 個増やす、それだけ。

4実例 1 — 画面遷移

canvas-ui-sample.js の画面遷移。if/else ゼロ。

const SCREEN_TRANSITIONS = [ { from: 'title', input: 'start', to: 'playing' }, { from: 'playing', input: 'pause', to: 'pause' }, { from: 'playing', input: 'die', to: 'gameover' }, { from: 'pause', input: 'resume', to: 'playing' }, { from: 'pause', input: 'quit', to: 'title' }, { from: 'gameover', input: 'retry', to: 'playing' }, { from: 'gameover', input: 'title', to: 'title' }, ]; function reduceScreen(state, input) { return SCREEN_TRANSITIONS.find(t => t.from === state && t.input === input)?.to ?? state; }

遷移の 全ケースが 1 箇所にある。AI が読むのもこの 7 行だけ。
遷移を追加する = 行を 1 つ増やす。制御フローを書き換える必要なし。

7なぜこれが ai-desk の核か

Constraint Folding は「テクニック」ではない。計算の単位そのもの。

(state, command) => newState
この単一の式が、ai-desk のすべてに現れる。
  • L3 Logic 層 (Bible §2): 純粋 Reducer = まさにこの式
  • Twin (§4.5): 同じ式を別実装で再計算 → 突合せて検証
  • Event Sourcing (§5): state = events を全部この式で reduce した結果
  • 状態遷移: 上記の例の通り
  • 網羅テスト: 全可能 (state, command) で reduce して検証
  • 逆引き: 「このsを得るには、どの (state, command) があり得るか」

→ ai-desk が 5 公理から導出された方法論というより、1 つのプリミティブから導出された体系。

8これが可能にすること

網羅テスト

全可能世界を列挙してあるので、テストは「全行を回す」だけ。漏れが原理的に発生しない。

逆引き

出力 → 入力の検索が可能 (filter の方向を変えるだけ)。分岐ツリーでは原理的に不可能。

静的検証

「すべての state に対して入力が定義されているか」をデータの完全性で検査できる。

可視化

データを graph として描画可能。状態遷移図が自動生成される。

AI による拡張

「新しい状態を追加して」 = AI が行を 1 つ追加する。コード書き換えゼロ、データ追記のみ。

仕様 = 実装

仕様書とコードが分離しない (データ自体が両方)。ドキュメント同期問題が消える。

9なぜ AI と親和性が高いか

分岐 = 認知の分裂。畳み込み = 認知の集約。

  • 分岐コードを読む AI: 各 if/else を順に追い、メンタルモデルを構築。10 段ネストで context 食い尽くし
  • 畳み込みコードを読む AI: 1 つの table を見て全パターン一望。reduce 関数 1 つで挙動完全把握
  • 分岐は 制御フロー = 時系列的、AI は時系列追跡が苦手
  • 畳み込みは データ = 空間的、AI は同時並行的処理が得意

Bible §0.0 「認知非対称性」の最も具体的な現れ。分岐を畳み込みにする = 人間のためのコードを AI のためのコードにする。

10適用範囲

強く効く領域

  • 有限離散状態 (UI 画面遷移 / ゲームロジック)
  • 業務ルール (税計算 / 割引判定 / 送料)
  • 権限・アクセス制御 (RBAC)
  • 状態機械 (workflow / approval flow)
  • 条件分岐の多いビジネスロジック

効かない領域

  • 連続値の物理計算・補間
  • リアルタイム描画ループ (60fps の中身)
  • 木探索・グラフ探索 (将棋 AI の読み等)
  • 機械学習の forward pass

アクションゲーム全体のバグ件数の 50-60% がこの構造で構造的に絶滅する (PROMPT_constraint_folding 測定値)。
